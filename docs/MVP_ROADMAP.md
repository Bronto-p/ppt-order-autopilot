# MVP Roadmap

## Phase 0: 框架

目标：

- 先完成隐私保护，确保真实 `orders/` 和运行时 `ledgers/*.jsonl` 不会进入 git。
- 建立目录、状态机、配置、模板和 skill 契约。
- 提供订单初始化工具。
- 提供硬门槛校验工具。

完成标准：

- 能创建标准订单目录。
- 能校验聊天采集、需求提取、决策和交付门槛。

## Phase 1: 询单和回复检查

目标：

- 固定时间向白名单客服发送固定询单。
- 按 2/5/10/30/60/120 分钟检查回复。
- 所有动作写入账本。

硬边界：

- 只允许白名单联系人。
- 只允许固定询单话术自动发送。
- 其他消息必须人工确认。

## Phase 2: 聊天采集

目标：

- 打开客服发来的聊天记录。
- 回到顶部。
- 重叠截图。
- OCR。
- 建立消息索引。
- 下载附件。
- 生成覆盖报告。

完成标准：

- 顶部锚点和底部锚点存在。
- 相邻屏幕重叠检查通过。
- 附件有 hash 和来源消息。

## Phase 3: 需求提取和接单建议

目标：

- 生成 `requirements.json`。
- 生成 `missing_questions.md`。
- 生成 `conflicts.md`。
- 生成 `decision.md`。
- 生成 `pending_approval.md`。

完成标准：

- 每个字段有证据。
- 缺失必填项会阻塞。
- 冲突会阻塞。
- 不会自动接单。

## Phase 4: PPT 生产接入

目标：

- 从订单需求生成 `production_contract.json`。
- 对接现有 `ppt-complete-workflow` 或新的生产 core。
- 保留全页图、页面 worker、PPTX/PDF 输出和截图质检。

调整方向：

- 降低重型计划依赖。
- 样稿改成状态机分支，不是固定 gate。
- 生产层只吃干净 contract。

## Phase 5: 交付和改稿闭环

目标：

- QA 通过后生成交付包。
- 交付话术进入人工确认。
- 改稿反馈结构化成 revision request。

完成标准：

- 模糊反馈会追问。
- 影响页码明确。
- 修改类型明确。
- 交付仍然需要人工确认。
